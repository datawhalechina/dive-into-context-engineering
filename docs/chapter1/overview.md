# 第一章：概览 —— 从提示词到自主智能体架构的范式跃迁

在与大型语言模型（LLM）交互的早期，我们往往认为只要“把话说清楚”，就能得到想要的结果。然而，随着 AI 应用逐渐深入到复杂的真实业务场景中，开发者们发现，仅靠单一的指令输入已经无法驾驭大模型在长周期、多跳推理任务中的不确定性。

为了突破大模型的“黑盒”限制，AI 工程师们的视角正在经历一次深刻的范式转移：从单纯的**提示词工程（Prompt Engineering）**，走向系统化的**上下文工程（Context Engineering）**，并最终迈向由多智能体协作构成的**框架工程（Harness Engineering）**。

## 1. 起点：提示词工程与“概率分布操控”
提示词工程是我们驾驭大模型的基础，它绝不仅仅是沟通技巧，而是一种大模型时代的全新“编程范式”。

* **本质是接口设计：** 提示词工程本质上是在控制模型行为的输入接口设计问题，通过自然语言调用具有千亿参数的概率引擎。
* **操纵概率分布：** 大模型的底层原理是下一个词预测，提示词工程的作用就是改变条件输入，从而精准收敛和定向操控输出的概率分布，让正确的 Token 出现的概率逼近 100%。
* **工程化局限性显现：** 当业务演进到需要处理长文档对比、实时查询数据库或复杂的错误重试逻辑时，纯粹的提示词工程会暴露出不稳定性、脆弱性以及状态管理缺失等痛点。把所有希望寄托在写一个完美的万能 Prompt 上是不切实际的。

## 2. 演进：上下文工程与“动态注意力调度”
因为纯提示词工程无法解决知识的动态更新和复杂逻辑编排问题，行业的前沿正在发生范式转移，走向上下文工程。如果说提示词工程的重点是“如何表达”，那么上下文工程的核心焦点则变成了“如何统筹”。

* **注意力预算的管理：** 模型就像人类的短期记忆一样，拥有一个有限的注意力预算，随着 Token 数量的增加，模型准确回忆信息的能力会下降，产生上下文腐败（Context Rot）。
* **将上下文视为稀缺资源：** 传统的做法是把所有相关信息塞进提示词，而上下文工程的核心哲学是信息密度最大化，利用压缩、选择和过滤机制，只保留信息密度最高的内容。
* **渐进式披露（Progressive Disclosure）：** 这是一个极其关键的设计哲学，主张不要一次性把所有信息都塞给模型，而要只在合适的时机，向它暴露当下真正需要的信息。

## 3. 终局探索：框架工程与“自主智能体循环”
当我们解决了单次对话的信息调度问题后，如何让模型在无人类干预的情况下，执行长达数小时甚至更久的复杂任务？这就需要引入受 GAN（生成对抗网络）启发的框架工程。

* **解决“上下文焦虑”：** 随着上下文窗口的填满，模型在冗长的任务中往往会失去连贯性，甚至因为接近上下文限制而表现出上下文焦虑，过早结束工作。
* **解耦生成与评估：** 当被要求评估自己产生的工作时，智能体往往会自信地赞美这些工作，即使质量平庸。将执行工作的智能体与评判工作的智能体分离开来，被证明是解决自我评估偏差的有力手段。
* **多智能体协作：** 一个成熟的长运行框架通常包含多个智能体角色，例如将用户提示扩展为产品规格的计划者（Planner）、以冲刺方式实现功能的生成器（Generator），以及像用户一样测试应用程序并根据标准打分的评估器（Evaluator）。这种生成器-评估器循环能够自然地映射到软件开发生命周期中，推动模型在复杂的全栈任务中产出极高质量的结果。

## 结语
从精心雕琢一段静态的系统提示，到构建动态调度知识库的 RAG 系统，再到编排多个智能体进行自我攻防与迭代，AI 工程师的武器库正在经历指数级的扩充。在这场演进中，唯一不变的真理是：**我们不再只是“向 AI 提问”，而是在“为 AI 搭建工作台”。**
